## 1. 背景与战略定位

### 1.1 问题陈述

当前企业 AI 应用普遍停留在「聊天机器人」层面：用户需要主动切换工具、手动粘贴数据、记忆每个智能体的能力边界。这带来三个核心痛点：

| 痛点 | 表现 | 业务影响 |
|------|------|----------|
| 能力孤岛 | 各部门各自维护独立 AI 工具，能力无法复用 | 重复建设成本高，ROI 低 |
| 数据断层 | AI 只能读用户粘贴的内容，无法直连企业系统 | 使用摩擦大，真实使用率低 |
| 治理缺失 | 无法统一管理谁在用什么 AI、效果如何 | 数据合规风险，无法量化 AI 价值 |

### 1.2 产品定位

数字员工模块将万卷平台从**「AI 工具集」升级为「AI 员工团队」**：

> 企业可以像管理真实员工一样管理 AI —— 有职位（角色）、有部门（业务域）、有绩效（评分/热度）、有入职流程（创建向导）、有工作记录（调用日志）。

**核心差异点**：

1. **域级隔离**：不同业务线的 AI 能力和知识库物理隔离，满足合规要求
2. **MCP 直连**：数字员工可直读数据库、飞书、CRM，而非依赖用户手动输入
3. **闭环评分**：用户每次对话后可评分，评分实时反映在管理台，形成 AI 质量 PDCA 闭环
4. **多入口部署**：同一员工可同时以 H5、elink、小程序、API、工作流节点形式提供服务

### 1.3 战略价值

```
短期（0-6个月）：提升平台粘性，增加企业内 AI 使用深度
中期（6-18个月）：成为企业 AI 治理基础设施，建立替换成本
长期（18个月+）：数字员工数量和质量成为平台核心竞争壁垒
```

## 2. 用户与场景

### 2.1 用户角色定义

#### 角色一：企业 AI 管理员（Admin）
**画像**：IT 部门或数字化转型团队，负责平台整体配置
**核心诉求**：我需要知道公司的 AI 资产总览，能统一管控，有问题能快速定位
**使用频率**：每周 2-3 次，主要查看数据大盘和处理告警

#### 角色二：业务域负责人（Domain Owner）
**画像**：法务总监、财务 BP、HR 总监等部门管理者
**核心诉求**：我要给我的部门配置专属 AI，只能看和用我自己域的内容，不想和其他部门混在一起
**使用频率**：每月配置 1-2 次，日常关注报表

#### 角色三：数字员工创建者（Builder）
**画像**：部门内的业务骨干或运营，懂业务但不懂代码
**核心诉求**：我要快速创建一个适合我业务场景的 AI 助手，不要让我写代码
**使用频率**：创建期每天使用，上线后每周维护

#### 角色四：普通用户（End User）
**画像**：各部门普通员工
**核心诉求**：我只想找到能帮我干活的 AI，用完给个反馈，不想管其他的
**使用频率**：每日使用，高频

### 2.2 核心用户故事

```
作为 业务域负责人，
我希望 能���我的部门创建专属数字员工，并绑定部门独有的知识库，
以便 员工得到的回答基于我们部门的实际制度，而不是通用知识。

作为 数字员工创建者，
我希望 通过4步引导完成员工配置（基础信息→AI能力→部署→测试），
以便 不需要懂 Prompt 工程也能创建出可用的数字员工。

作为 普通用户，
我希望 在前台直接选择适合我任务的数字员工，用完可以打分，
以便 好的员工能被更多同事发现，差的能得到改进。

作为 AI 管理员，
我希望 在工作台一页看到全公司 AI 调用量、运行状态和待处理事项，
以便 不需要登多个后台就能掌握整体 AI 运行健康度。
```

## 3. 产品目标与成功指标

### 3.1 目标体系（OKR）

**Objective**：让企业 AI 从「单点工具」进化为「可管理的 AI 员工团队」

| Key Result | 指标 | 基线 | 目标（6个月） | 测量方式 |
|-----------|------|------|--------------|---------|
| KR1 | 企业平均数字员工数 | 0 | ≥ 5 个/企业 | 数字员工库统计 |
| KR2 | 员工平均月调用量 | - | ≥ 500 次/员工 | 调用日志 |
| KR3 | 用户评分收集率 | 0% | ≥ 30% | 评分提交数/对话数 |
| KR4 | 员工平均用户评分 | - | ≥ 4.0 分 | 评分均值 |
| KR5 | 域级 MCP 绑定率 | 0% | ≥ 40% | 绑定 MCP 的员工比例 |

### 3.2 反指标（不应追求的）

- **不**以数字员工数量增长为核心目标（防止为创建而创建的空壳员工）
- **不**以页面 PV 衡量（应以实际调用量和任务完成率衡量真实价值）

## 5. 功能详细设计

### 5.1 工作台

#### 5.1.1 欢迎横幅

**功能**：提供身份感��的个性化欢迎，快速操作入口

| 元素 | 说明 | 备注 |
|------|------|------|
| 欢迎语 + 日期 | 固定文案 + 系统日期 | 后期可接入个性化打招呼 |
| 组织层级标签 | 用户在工作台配置中选择（集团/子公司/部门等） | 影响数据范围筛选 |
| 岗位角色标签 | 管理层/一线员工/运维人员 | 影响默认模块显示 |
| 运行员工数 | 来自 Store 实时数据 | 点击跳转员工库 |
| 「配置工作台」 | 打开配置 Drawer | |
| 「创建员工」 | 跳转员工库并打开创建 Modal | |
| 「员工库 →」 | 跳转员工库 | |

#### 5.1.2 统计看板

**交互规则**：
- 默认展示「本周」数据
- 切换时段（今日/本周/本月/本季）时，4 张卡片和趋势图同步更新
- 「调用量」和「运行中员工数」从共享 Store 读取实时值
- 每张卡片右下角展示同比变化（箭头 + 百分比），绿色=正向，黄色=负向

**需求细节**：
- 「调用量」卡：≥ 1000 时显示 `xx.xk` 格式
- 「完成率」卡：数值 < 90% 时用黄色标注，提示需关注

#### 5.1.3 我的收藏

**核心交互**：
- 横向滚动卡片列表，最右侧固定「+ 添加收藏」占位卡
- 单张卡片右上角「✕」可直接取消收藏，无需进入管理 Modal
- 点击「管理收藏」打开 Modal，支持搜索 + 星标切换
- 收藏状态持久化（本期存 localStorage，后期同步账号）

**边界情况**：
- 被收藏员工下线/归档时，卡片显示「已暂停」状态但不自动移除，等待用户手动取消

#### 5.1.4 待处理事项

每条事项包含：优先级（高/中/低）、事项描述、标签、时间戳、操作按钮

**上下文导航规则**：点击操作按钮直接跳转至对应页面（员工库/域管理），且目标页面自动高亮相关员工或开启对应 Drawer，减少用户二次寻找成本。

> **设计说明**：「待处理事项」本期为静态 Mock，需后端提供事件推送接口（见第 11 节依赖）

#### 5.1.5 工作台配置 Drawer

| 配置项 | 类型 | 说明 |
|--------|------|------|
| 组织层级 | 单选 | 影响工作台数据范围（集团/子公司/部门/市局/区县） |
| 岗位角色 | 单选 | 管理层/一线员工/运维人员 |
| 模块显示 | 开关 + 排序 | 可隐藏任意模块，可通过上/下按钮调整顺序 |

### 5.2 数字员工库

#### 5.2.1 员工状态机

```
草稿(draft) ──→ 测试中(testing) ──→ 运行中(published)
     ↑                                      ↓
     └──────────────── 已暂停(paused) ←──────┘
                              ↓
                         已归档(archived)
```

**状态转换权限**：
- draft → testing：创建者
- testing → published：域管理员审批（本期简化为创建者直接发布）
- published → paused：创建者或域管理员
- paused → published：创建者或域管理员
- 任意 → archived：域管理员

#### 5.2.2 创建向导

**Step 0（全局）**：头像根据名称哈希自动生成渐变色，无需用户上传

**Step 1 - 基础信息**：

| 字段 | 必填 | 说明 |
|------|------|------|
| 员工名称 | ✓ | 最大 20 字，实时预览头像 |
| 所属部门 | ✓ | 下拉选择，影响默认知识库推荐 |
| 业务域 | ✓ | 决定该员工归属哪个业务域 |
| 角色定位 | ✓ | 自由填写，写入 System Prompt 占位符 |
| 功能描述 | 可选 | 最大 200 字，在员工卡片和前台展示 |

**Step 2 - AI 能力**：

*模型选择*：卡片式选择，展示模型名称、适用场景标签、说明文字。域级默认模型可在员工级覆盖。

*角色提示词*：
- 提供标准模板（含占位符 `{员工名称}` `{部门}` `{角色定位}`）
- 支持「重置模板」一键还原
- 字���数 < 30 时在 Step 4 配置核查中标记警告

*知识库绑定*：列表式开关，继承域级配置，可在员工级覆盖

*技能（Skills/工作流）*：列表式开关，默认开启「网络搜索」和「飞书文档」

*MCP Server 绑定*：
- **这是区分「数字员工」和「聊天机器人」的核心能力**
- 绑定后员工可直接读写企业内部系统（数据库/飞书/GitHub/CRM/ERP）
- 每个 MCP 有独立开关，绑定后在员工卡片上显示 MCP 图标

**Step 3 - 部署配置**：

*回复风格*：正式严谨 / 专业友好 / 简洁直接（影响 System Prompt 中的风格指令）

*访问权限*：
- 仅创建者（私有）
- 部门内（部门成员可见）
- 全公司（所有人可见）

*接入渠道*（多选）：

| 渠道 | 类型 | 说明 |
|------|------|------|
| H5 网页 | 主渠道 | 电脑/移动端浏览器直接访问 |
| elink 嵌入 | 主渠道 | 政企内部办公系统/政务平台嵌入 |
| 小程序 | 主渠道 | 政企专属 APP 小程序 |
| REST API | 扩展 | 标准 HTTP 接口 |
| 工作流节点 | 扩展 | 嵌入万卷工作流 |
| 飞书机器人 | 扩展 | 飞书群机器人 |
| 钉钉机器人 | 扩展 | 钉钉群机器人 |

**Step 4 - 测试发布**：

布局分左（对话测试区）右（配置核查）两栏：

*对话测试区*：
- 与真实前台一致的消息气泡 UI
- AI 默认发一条打招呼消息（根据名称和角色动态生成）
- 用户可发送测试消息，AI 以当前 Prompt 回复（Mock 回复，非真实 LLM 调用）
- 测试模式 Tag 明确标注

*配置核查*：
- 自动检查：员工身份、AI 模型、提示词长度、接入渠道
- 全部通过：绿色「配置完整，可以发布」
- 有项目待完善：黄色警告，仍可保存草稿

操作：「保存草稿」「发布上线」（两个按钮并排，用户明确选择）

#### 5.2.3 版本管理

- 每次「发布上线」自动生成版本快照
- 历史版本支持「回滚」，回滚前需二次确认
- 当前版本高亮标注，不可回滚

#### 5.2.4 多入口接入 Drawer

包含两个核心部分：

**API Key 管理**：
- 每个员工有独立 API Key，格式：`wj_sk_••••{id后4位}`
- 展示今日调用、本月调用、速率限制三项指标
- 支持「重新生成 Key」（旧 Key 24 小时内仍有效，给调用方迁移时间）

**各渠道接入代码**：
- H5 网页：直接访问 URL
- elink 嵌入：`<iframe>` 代码片段
- 小程序：配置参数（app_id、entry_type、platform）
- REST API：Endpoint + Authorization 示例
- 工作流节点：agent_id + call_mode 配置

每个渠道都有一键复制按钮。

### 5.3 业务域管理

#### 5.3.1 业务域的核心概念

**业务域 = AI 组织的部门单元**

类比真实公司结构：
```
公司（万卷租户）
├── 法务合规域    → 法务部所有 AI 能力的集合
├── 人力资源域    → HR 部所有 AI 能力的集合
└── 财务管理域    → 财务部所有 AI 能力的集合
```

**域的价值**：
1. **知识隔离**：法务的合同库不会被财务员工的 AI 读取
2. **能力继承**：域内员工自动继承域级知识库和模型配置，减少重复设置
3. **权限边界**：跨域调用需经域管理员授权

#### 5.3.2 隔离级别

| 级别 | 规则 | 适用场景 |
|------|------|---------|
| 严格隔离 | 数据完全隔离，跨域访问须逐次授权 | 法务、财务等高合规部门 |
| 标准隔离 | 域内完全共享，跨域按需申请 | 大多数业务部门 |
| 开放共享 | 全平台所有员工均可调用本域能力 | 基础设施类域（如通用搜索） |

#### 5.3.3 域卡片设计

每张域卡片展示：
- 颜色标识条（顶部 6px 色条，各域独立颜色）
- 域图标（Emoji，可选12种）+ 域名称 + 分类 Tag
- 运行状态 Badge
- 4项指标格子：员工数 / 调用量 / 完成率 / 复用率
- 任务完成率进度条（≥90% 绿色，< 90% 黄色）
- 标签 + 隔离级别标注
- 操作：编辑 / 配置 / 统计 / 删除

#### 5.3.4 域配置 Drawer（三 Tab）

**Tab 1 - 能力隔离**：卡片式选择隔离级别，附说明文字和建议用法

**Tab 2 - 资源调配**：
- 域专属 AI 模型（可被员工级覆盖）
- 资源使用进度条：API 调用配额 / 知识库存储 / 并发连接数
- > 80% 时进度条变红，提示扩容

**Tab 3 - 知识库配置**：
- 列表展示所有知识库，每项有读/写权限标注 + 开关
- 蓝色说明条：「域级配置后，本域所有数字员工默认继承访问权限，可在员工级别单独覆盖」

#### 5.3.5 域统计 Drawer

- 时段选择（本日/本周/本月/本季），2×2 指标卡
- 近 7 天调用趋势柱状图（CSS 实现，无需图表库）
- 域间横向完成率对比，当前域高亮显示

### 5.4 前台交互与评分（完整链路）

#### 5.4.1 数字员工入口

位置：前台侧边栏，在「智能体广场」下方，固定显示「🤖 数字员工」

点击后展示**员工选择面板**：
- 搜索框（按名称/部门过滤）
- 员工卡片网格（只展示 published 状态）
- 每张卡片：头像渐变 / 名称 / 部门·域 / 描述（2行截断）/ 用户评分星级 / 调用量 / 「开始对话 →」

**员工选择 → 对话创建**：
- 点击员工卡片，自动创建以该员工为对象的新会话
- 该员工作为 agent 驱动对话（复用现有对话引擎）
- 侧边栏出现黄色「⭐ 为「xxx」评分」快捷条

#### 5.4.2 评分弹窗（SessionRatingModal）

**触发时机**：
- 用户主动点击侧边栏「为 xxx 评分」按钮（随时可触发）
- 本期不做自动弹出（防止打断用户流程）

**弹窗内容**：

1. **员工信息区**：头像 + 名称 + 部门 + 引导语「本次对话体验如何？」

2. **5 星评分区**：
   - 星星 hover 时放大 1.2 倍，有动效
   - hover/选中时底部显示文字标签（很差/较差/一般/满意/非常满意）

3. **快捷标签区**（选星后出现）：
   - 5星：「回答准确」「效率极高」「非常有帮助」「超出预期」
   - 3星：「基本可用」「部分准确」「有待改进」
   - 1星：「完全无用」「答非所问」「误导信息」
   - 标签按分值动态联动，多选

4. **文字评价区**：TextArea，选填，最大 200 字，showCount

5. **操作按钮**：「跳过」（关闭，不提交）「提交评分」（选星后激活）

**提交后**：
- 弹窗切换为感谢页（动画 🎉）
- 展示本次评分星级（星星高亮）
- 说明「评分已实时更新员工综合评分」
- 3秒后自动关闭或用户点击「完成」

#### 5.4.3 评分计算规则

```
新综合评分 = Round((原评分总和 + 本次评分) / 新评分总人次, 1)

新热度 = Min(100, Round(当日调用次数 / 历史峰值日调用次数 × 100))

注意事项：
- 单个用户同一会话只能评分一次（以 sessionId 去重）
- 综合评分保留一位小数
- 热度值上限 100，防止数值异常
```

#### 5.4.4 管理台联动

评分提交后，通过 `employeeStore.submitRating()` 写入共享 Store，所有订阅了 Store 的组件（员工库、工作台）在当前会话内实时刷新，无需页面重载。

## 8. 非功能需求

### 8.1 性能

| 指标 | 要求 |
|------|------|
| 工作台首屏加载 | < 1.5s（FCP） |
| 员工库卡片渲染（50张） | < 500ms |
| 评分提交响应 | < 200ms（前端 Store 同步），< 2s（后端落库） |
| 创建向导步骤切换 | < 100ms |

### 8.2 兼容性

- 浏览器：Chrome 90+, Edge 90+, Safari 14+, Firefox 88+
- 屏幕分辨率：最小支持 1280×800
- 移动端：员工库和工作台需在 375px 宽度下可用（响应式降级：3列 → 1列）

### 8.3 可用性

- 关键操作（删除域、归档员工）需二次确认
- 表单校验错误需在字段下方实时提示，不能只在提交时统一报错
- 加载状态需有骨架屏或 Spinner，不能空白

### 8.4 数据安全

- MCP 连接凭证（数据库密码、API Token）服务端加密存储，前端只展示脱敏后的标识
- 严格隔离域的员工调用日志不能被其他域用户查看
- API Key 只在生成时完整展示一次，之后展示脱敏版本

## 12. 智能管理助手（AI Ops Assistant）

> **设计背景**：管理员和域管理者日常需要在多个模块间跳转才能获取运营全貌。引入自然语言交互层，让管理者以对话方式直接查询运行状态、生成报告、获取洞察，减少认知负担，提升决策效率。

---

### 12.1 功能定位

| 维度 | 说明 |
|------|------|
| **是什么** | 嵌入工作台的 AI 对话助手，专为数字员工运营管理场景设计 |
| **不是什么** | 不是通用聊天机器人；不处理与数字员工管理无关的问题 |
| **目标用户** | 超级管理员、域管理员（不面向普通用户） |
| **核心价值** | 将「多页跳转 + 手动筛选 + 人工汇总」压缩为「一句话问答」 |

---

### 12.2 入口设计

**方案：工作台悬浮入口 + 右侧滑出面板**

```
工作台页面
└── 右下角悬浮按钮（固定定位）
    ├── 图标：✨ 或 💬
    ├── 文字：AI 助手
    └── 点击后 → 右侧滑出 Drawer（宽 400px）
        ├── 顶部：助手标题 + 当前数据范围标注
        ├── 中部：对话消息区（气泡式）
        ├── 快捷提问卡（首次打开展示）
        └── 底部：输入框 + 发送按钮
```

**入口触发规则**：
- 工作台页面常驻显示悬浮按钮
- 员工库、域管理页面同样展示，但助手上下文自动切换为对应模块
- 有未读洞察时，悬浮按钮显示红点提醒

---

### 12.3 核心能力矩阵

#### A. 查询类（Query）

用户以自然语言描述问题，助手实时查询当前数据返回结构化答案。

| 示例问句 | 返回形式 |
|----------|----------|
| 法务合规助手今天被调用了多少次？ | 数字卡片 + 同比变化 |
| 各业务域本周调用量排名 | 排行榜列表（可点击跳转） |
| 当前哪些员工处于测试中？ | 员工卡片列表 |
| 评分低于 4.0 的员工有哪些？ | 警示卡片列表 + 「去处理」跳转 |
| 智能客服域的完成率趋势怎么样？ | 迷你折线图 + 文字解读 |
| 今天有没有调用异常？ | 告警摘要 or「一切正常」确认消息 |

#### B. 报告类（Report Generation）

助手根据指令生成结构化运营报告，支持预览和导出。

| 报告类型 | 触发语句示例 | 包含内容 |
|----------|-------------|----------|
| 日报 | 「生成今日运营日报」 | 调用量、活跃员工数、异常事件、值得关注的变化 |
| 周报 | 「生成本周数字员工周报」 | 各域对比、Top 3 员工、评分变化、待处理事项 |
| 月报 | 「帮我出一份 3 月的月度分析」 | 月度趋势、同比环比、域健康度、改进建议 |
| 域报告 | 「生成法务域的运营报告」 | 域内员工详情、调用量分布、知识库使用情况 |
| 员工报告 | 「财务报表助手的详细报告」 | 单员工全维度：调用/评分/版本/告警历史 |

**报告交互流程**：
```
用户输入「生成本周周报」
    → 助手回复「正在生成，预计 3 秒...」（骨架屏动效）
    → 报告卡片展开（富文本，含图表缩略图）
    → 底部操作栏：「复制文本」「导出 PDF」「发送至飞书」
    → 用户可追问：「重点解释一下调用量下降的原因」
```

#### C. 洞察类（Proactive Insights）

助手主动推送异常发现，无需用户主动提问。

| 洞察类型 | 触发条件 | 消息样式 |
|----------|----------|----------|
| 评分下滑告警 | 员工评分 7 天内下降 > 0.5 | 🔴 警示卡 + 「查看评分明细」 |
| 调用量骤降 | 某员工调用量较上周同期下降 > 30% | 🟡 注意卡 + 可能原因推测 |
| 待处理积压 | 待处理事项 > 5 条且 > 24 小时未处理 | 🔴 催办提醒 |
| 员工表现优秀 | 员工连续 7 天评分 ≥ 4.8 | 🟢 表扬推送（可一键分享至企业群） |
| 资源接近上限 | 某域 API 配额使用率 > 80% | 🟡 扩容建议 |

#### D. 操作类（Action Triggers）

助手可代为执行管理操作，**所有操作均需二次确认**，敏感操作额外显示影响范围。

| 操作示例 | 确认机制 |
|----------|----------|
| 「暂停财务报表助手」 | 弹出确认卡：员工名称 / 当前状态 / 暂停后影响用户数 / 确认按钮 |
| 「把代码审查助手切换到测试状态」 | 确认卡 + 显示当前版本 |
| 「给智能客服助手添加一条待办备注」 | 直接执行，展示写入结果 |
| 「帮我把本周周报发给张总」 | 需要先绑定通讯录，首次使用引导授权 |

> **安全边界**：助手不可执行删除员工、删除域、重置 API Key 等不可逆操作，此类操作仅返回「请前往对应页面手动操作」并附跳转链接。

#### 对话气泡规范

| 元素 | 规范 |
|------|------|
| 用户气泡 | 右对齐，紫色背景，白色文字 |
| 助手气泡 | 左对齐，白色背景，深色文字，支持富文本（表格/列表/图表） |
| 数据卡片 | 嵌入气泡内，带边框，关键数字高亮显示 |
| 操作按钮 | 气泡底部附加，描边样式，点击触发对应流程 |
| 加载状态 | 三点动效（...），超过 3 秒显示「数据量较大，稍候...」 |

#### 上下文保持规则

- 同一会话内保持 10 轮上下文记忆
- 用户可说「就上一个员工」「换成上个月」等指代，助手自动解析
- 切换工作台 Tab 后，助手上下文不清空，但提示「您已切换到员工库，我可以帮您查询员工详情」

### 12.9 反模式（避免设计陷阱）

1. **不要做成通用聊天机器人**：严格限定在数字员工管理场景，拒绝无关问题时给出友好引导（「我专注于数字员工管理，您可以问我关于员工运营的任何问题」）
2. **不要让操作无确认直接执行**：所有写操作必须展示影响范围并由用户二次确认
3. **不要在数据不足时编造答案**：数据缺失时明确告知「当前数据不足，建议查看…」，不要用模糊语言掩盖
4. **不要让报告变成纯文字堆砌**：报告必须包含可视化数据摘要（表格/迷你图），纯文字报告用户不会读
5. **不要忽略权限边界**：任何查询都先校验权限，越权查询返回「您无权查看该域数据」而非空数据
