## 1. 背景与目标

### 1.1 背景

企业在 AI 落地过程中面临三大痛点：

- **配置门槛高**：绑定知识库、调试 Prompt、对接内部系统需要较强技术能力，业务人员难以独立完成
- **管理碎片化**：AI 能力散落在不同工具，缺乏统一生命周期管理（版本、权限、监控）
- **价值难量化**：缺乏面向管理层的可视化度量体系，AI 投入效益无从衡量

数字员工模块旨在解决上述问题，提供 **"配置即服务、使用即交付"** 的完整闭环。

### 1.2 一期目标

> **核心命题**：非技术业务人员能够独立完成一个数字员工的从零到发布，任意员工能够自然地使用数字员工完成业务任务。

| 目标维度 | 具体目标 |
|---------|---------|
| 配置效率 | 非技术人员完成创建 → 配置 → 发布全流程 ≤ 45 分钟 |
| 使用体验 | 前台对话满意度评分均值 ≥ 4.2 星 |
| 功能完整性 | 配置侧 P0 功能 100% 交付；使用侧 P0 功能 100% 交付 |
| 数据互通 | 使用侧评分数据实时回流至配置侧统计，延迟 ≤ 3s |

### 1.3 与主产品的关系

数字员工模块作为万卷企业 AI 平台的**核心业务模块**独立运行，通过 `#digital-employee-*` hash 路由体系接入主平台导航，数据层通过 `employeeStore` 发布-订阅机制与其他模块共享状态。

## 2. 一期范围定义

### 2.1 纳入一期

| 模块 | 定位 | 核心价值 |
|------|------|---------|
| **员工库（Library）** | 数字员工全生命周期管理 | 创建、编辑、发布、版本管理、接入管理 |
| **配置向导（Wizard）** | 员工 AI 能力与部署配置 | 模型、提示词、知识库、技能、渠道 |
| **调试预览** | 发布前验证 | 配置核查 + Mock 对话预览 |
| **前台（Frontend）** | 数字员工使用界面 | 对话、工具调用可视化、评分反馈 |
| **工作台（Workbench）** | 运营数据大盘 | KPI 统计、收藏快访、域员工概览 |
| **业务域（Domain）** | 基础治理能力 | 域 CRUD、隔离级别配置 |
| **定时任务（Scheduled）** | 自动化执行 | 任务创建、执行日志查看 |
| **360° 画像（Profile）** | 价值度量 | 六维能力评估、用户评价汇总 |

### 2.2 一期不纳入（明确 Out of Scope）

| 功能 | 原因 | 规划版本 |
|------|------|---------|
| 知识库上传/管理 UI | 依赖知识库模块独立建设 | v1.1 |
| 真实大模型 API 对接 | 一期以 Mock 流式输出替代 | v1.1 |
| 跨域访问申请/审批流 | 需要流程引擎支撑 | v1.2 |
| 移动端适配 | 先完成 PC 端体验闭环 | v1.2 |
| 语音/数字人交互 | 依赖 TTS/ASR/数字人中台 | 待定 |
| 多语言（en-US） | 优先服务国内用户 | v1.2 |
| 员工间自主通信（MAS） | 架构复杂度高 | v2.0 |

## 3. 用户角色

### 3.1 角色定义

```
┌─────────────────────────────────────────────────────────────┐
│  配置侧角色                          使用侧角色               │
│                                                             │
│  ① 平台管理员                        ③ 普通员工（最终用户）    │
│     负责整体治理、域管理、配额分配                              │
│                                                             │
│  ② AI 配置员（业务+技术复合角色）     ④ 数据分析师              │
│     负责创建/维护数字员工                查看画像和大盘数据       │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 角色诉求 & 成功标准

**① 平台管理员**

| 诉求 | 优先级 | 验证标准 |
|------|-------|---------|
| 建立组织化的业务域体系，避免 AI 资产无序增长 | P0 | 创建业务域并为员工分配域归属 |
| 控制员工可见范围（私有/部门/全公司） | P0 | 权限设置生效，越权访问被拦截 |
| 查看平台整体使用数据 | P1 | 工作台展示有效的汇总统计 |

**② AI 配置员**

| 诉求 | 优先级 | 验证标准 |
|------|-------|---------|
| 用最少步骤创建一个可用的数字员工 | P0 | 单次完整创建流程 ≤ 45 分钟 |
| 能在发布前预览效果、核查配置完整性 | P0 | 配置核查清单全绿时方可发布 |
| 能对已发布员工进行迭代而不中断服务 | P0 | 新版本发布、历史版本可回滚 |
| 能灵活选择多种接入渠道 | P1 | 接入代码/链接可一键复制 |

**③ 普通员工**

| 诉求 | 优先级 | 验证标准 |
|------|-------|---------|
| 快速找到适合当前任务的数字员工 | P0 | 前台支持按域/名称过滤 |
| 与数字员工进行自然流畅的多轮对话 | P0 | 对话延迟 ≤ 3s（Mock 环境） |
| 知道 AI 在做什么（工具调用透明可见） | P1 | 工具调用步骤可展开查看 |
| 对不满意的回复给出反馈 | P1 | 评分数据实时同步至画像 |

**④ 数据分析师**

| 诉求 | 优先级 | 验证标准 |
|------|-------|---------|
| 直观量化每个数字员工的业务价值 | P1 | 360° 画像可查看六维评分+调用量 |
| 横向对比不同员工或域的表现 | P2 | 域统计支持完成率排序对比 |

## 4. 核心用户旅程

### 4.1 配置旅程：从零到发布一个数字员工

```
入口：员工库页面 → 点击「创建数字员工」

Step A  选择创建方式
   ├─ 空白创建：从模板快速开始 → 填写基础信息
   └─ AI 创建：输入自然语言描述 → AI 自动生成配置

Step B  基础信息（弹窗 Step 1）
   员工名称（必填）+ 所属部门（必填）+ 业务域（必填）+ 员工职责（选填）
   → 点击「下一步」进入全页配置向导

Step C  AI 能力配置（全页左侧）
   ① 选择 AI 模型（4 选 1）
   ② 编写角色提示词（System Prompt）
   ③ 绑定知识库（多选 Switch）
   ④ 启用技能（多选 Switch）
   ⑤ 绑定 MCP Server（多选 Switch）

Step D  部署配置（全页左侧，续）
   ⑥ 设置访问权限（全公司 / 仅自己 / 指定部门人员）
   ⑦ 选择接入渠道（H5 / elink / 小程序 + 更多）

Step E  调试预览（全页右侧，与 C/D 同步）
   配置核查清单（实时检查 5 项配置完整性）
   调试预览区（Mock 对话界面，展示员工欢迎语和建议问题）

Step F  发布
   顶部「发布」按钮 → message.success → 状态变为「运行中」
   员工出现在员工库列表，可被前台调用
```

**关键决策点与容错设计：**

| 节点 | 容错规则 |
|------|---------|
| 基础信息未完整 | 「下一步」按钮禁用，hovering 无响应 |
| Prompt 字符不足 30 | 配置核查标记 ⚠️，不阻止保存，但提示"内容过少" |
| 渠道未选择 | 配置核查标记 ⚠️，顶部发布按钮仍可点击（允许先发布再配置渠道） |
| 配置随时自动保存 | 左侧任意修改 800ms 后显示「✓ 已自动保存」 |

---

### 4.2 使用旅程：普通员工使用数字员工

```
入口：主导航「数字员工」前台页面

Step A  选择员工
   侧边栏员工列表 → 搜索/按域筛选 → 点击员工进入对话

Step B  发起对话
   输入框输入问题（支持 @提及员工、上传文件）
   → 发送 → 员工开始响应

Step C  查看响应
   ├─ 简单对话：文字回复直接展示
   ├─ 带工具调用：展示 ToolCall 进度（searching / calling / completed）
   └─ 复杂任务：展示规划 → 分发 → 执行 → 汇总 多阶段进度

Step D  对话管理
   历史记录自动保存到左侧会话列表
   支持新建会话 / 置顶会话 / 删除会话

Step E  评价反馈
   点赞/踩 → 单条消息反馈
   会话结束评分弹窗 → 整体满意度（1-5 星）
   → 数据实时回流到 employeeStore → 画像/工作台更新
```

## 5. 功能需求 · 配置侧

### 5.1 员工库（DigitalEmployeeLibrary）

#### 5.1.1 员工列表

**卡片视图**（默认，3 列网格）

| 功能项 | 优先级 | 需求描述 |
|-------|-------|---------|
| 头像渐变色 | P0 | 根据员工名称+部门+域哈希生成渐变色，保证同名员工视觉区分 |
| 状态 Badge | P0 | draft=灰/testing=蓝/published=绿/paused=橙/archived=红，与操作按钮联动 |
| 三指标展示 | P0 | 调用量（≥1000 时转 k 单位）/ 用户评分（⭐角标 + Tooltip 说明抽样人数）/ 活跃热度（近7日活跃率） |
| 访问权限图标 | P0 | 🔒仅创建者 / 👥部门内 / 🌐全公司，列表视图增加文字说明 |
| hover 浮起效果 | P1 | translateY(-2px) + boxShadow 加深 + 边框变紫，transition 200ms |

**列表视图**（表格，用户手动切换）

列：数字员工（头像+名称+部门·域·版本）/ 类型 / 状态 / 访问权限 / 调用量 / 评分 / 更新时间 / 操作

**筛选与搜索**

| 功能项 | 优先级 | 需求描述 |
|-------|-------|---------|
| 名称/描述搜索 | P0 | 输入即过滤，无防抖要求（数据量 ≤ 100） |
| 状态筛选下拉 | P0 | 空时显示全部，选中后与搜索条件 AND 关系 |
| 业务域 Tab | P0 | 「全部」+ 各业务域名称，颜色对应域颜色，点击单选过滤 |
| 结果计数 | P0 | 工具栏右侧显示「共 N 个」 |

#### 5.1.2 员工生命周期状态机

```
                   ┌──────────────────────────────┐
                   │                              │
[草稿 draft] ──→ [测试中 testing] ──→ [运行中 published]
      │                   │                    │      ↑
      │ 直接发布            └── 直接发布         │      │ 恢复
      │                                        ↓      │
      └──────────────────────────────→ [已暂停 paused]
                                              │
                            任意状态 ─────────↓
                                       [已归档 archived]（不可恢复）
```

| 状态转换 | 触发操作 | 约束条件 |
|---------|---------|---------|
| draft → published | 点击「发布」 | 无强制约束（允许配置不完整先发布） |
| testing → published | 点击「发布」 | 同上 |
| published → paused | 点击「暂停」 | 无 |
| paused → published | 点击「恢复」 | 无 |
| 任意 → archived | 点击「归档」 | Modal.confirm 二次确认，危险按钮样式 |

#### 5.1.3 行内操作

| 操作 | 显示条件 | 交互 |
|------|---------|------|
| 编辑 | 所有状态 | 打开全页编辑向导 |
| 版本 | 所有状态 | 打开版本管理 Drawer |
| 接入 | 所有状态 | 打开多入口接入 Drawer |
| 发布 | draft / testing | 点击直接发布，message.success |
| 暂停 | published | 点击直接暂停 |
| 恢复 | paused | 点击直接恢复 |
| 归档 | 非 archived | Modal.confirm 确认 |

**左侧：底部导航栏**

### 5.3 版本管理（Drawer，宽 480px）

| 功能项 | 优先级 | 需求描述 |
|-------|-------|---------|
| 版本列表 | P0 | 时间倒序，显示：版本号 / 更新日志 / 发布时间 / 发布者 |

## 8. 数据模型

### 8.1 核心实体关系

```
业务域 Domain 1..N
    └── 数字员工 DigitalEmployee
            ├── 版本 Version 1..N
            ├── 定时任务 ScheduledTask 0..N
            │       └── 执行日志 TaskRunLog 0..N
            └── 评价数据 Rating 0..N
                    └── 聚合为 Profile 六维分值
```

### 8.2 数字员工主体

```typescript
interface DigitalEmployee {
  // 唯一标识
  id: string;                    // 格式 de-xxx，系统自动分配

  // 基础信息
  name: string;                  // ≤ 20 字
  dept: string;                  // 来自 DEPARTMENTS 枚举
  domain: string;                // 来自 Domain 模块
  description: string;           // ≤ 200 字
  role: string;                  // 角色定位，如"法务合规顾问"
  type: '通用款' | '定制款' | '升级款';

  // 生命周期
  status: 'draft' | 'testing' | 'published' | 'paused' | 'archived';
  version: string;               // vX.Y.Z，发布时自动递增 patch

  // 权限控制
  scope: 'private' | 'dept' | 'company' | 'custom';
  accessTargets?: string[];      // scope='custom' 时生效，存部门/人员 ID

  // AI 能力配置
  model: 'chat-model' | 'gpt-4o' | 'qwen-max' | 'glm-4';
  systemPrompt: string;
  knowledgeBases: string[];      // kb1~kb5
  skills: string[];              // sk1~sk5
  mcpServers: string[];          // mcp1~mcp5
  responseStyle: 'formal' | 'friendly' | 'concise';
  channels: string[];            // h5 / elink / miniapp / api / flow / feishu / ding

  // 运营指标（由 Store 维护）
  callCount: number;
  score: number;                 // 1-5，保留1位小数
  heat: number;                  // 0-100
  ratingCount: number;
  ratingSum: number;
  recentCalls: number[];         // 近7日每日调用量

  // 时间
  updateTime: string;            // ISO 日期
}
```

### 8.3 评分数据

```typescript
interface SessionRating {
  employeeId: string;
  sessionId: string;
  score: number;                 // 1-5
  tags: string[];                // 快捷标签数组
  comment: string;               // 文字评价（可空）
  timestamp: string;
}
```

### 8.4 热度算法

```
heat = Math.min(100, Math.round( sum(recentCalls) / historicalPeak * 100 ))

其中：
- recentCalls = 近7日每日调用量数组
- historicalPeak = 历史单日最高调用量
- 结果取整，上限 100
```

### 8.5 状态存储（employeeStore）

```typescript
// 发布-订阅接口
class EmployeeStore {
  subscribe(fn: () => void): () => void   // 返回取消订阅函数
  getEmployees(): EmployeeRecord[]
  getEmployee(id: string): EmployeeRecord | undefined
  updateEmployee(id: string, patch: Partial<EmployeeRecord>): void
  submitRating(rating: SessionRating): void
  getStats(): { activeCount: number; totalCalls: number; avgScore: number; totalRatings: number }
}

// React 订阅模式（组件内使用）
const [, forceUpdate] = useState(0);
useEffect(() => {
  const unsub = employeeStore.subscribe(() => forceUpdate(n => n + 1));
  return () => { unsub(); };
}, []);
```

| 创建/发布成功 | `message.success('数字员工「{name}」已发布上线')` |

### B. 业务规则汇总

| 编号 | 规则 |
|------|------|
| R-01 | 员工 ID 格式：`de-XXX`，系统自动分配，不可重复、不可手动修改 |
| R-02 | 版本号格式：语义化版本 `vX.Y.Z`，每次发布自动递增 patch |
| R-03 | Heat 热度 = 近7日总调用量 / 历史单日峰值调用量 × 100，上限 100 |
| R-04 | 综合评分 = 系统监控×40% + 用户评价×35% + 任务指标×25%，保留1位小数 |
| R-05 | 已归档员工不可恢复运行，不可编辑，仅可查看历史记录 |
| R-06 | 重新生成 API Key 后，旧 Key 24小时内仍有效，之后自动失效 |
| R-07 | API 默认速率限制：100次/分钟，企业管理员可申请提升配额 |
| R-08 | 严格隔离模式下跨域访问：返回 403，日志记录拦截事件 |
| R-09 | 删除业务域前需将该域内所有员工迁移至其他域或归档 |
| R-10 | 定时任务 error 状态需手动介入修复后方可重新 active |
